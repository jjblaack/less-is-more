# Doc Templates · 三类核心文档模板

## 设计思想

**为什么只有这三类？**
- 项目长期维护中，文档最容易腐坏。腐坏的常见原因是"写得太细、更新太慢"。
- 代码本身就是最细、最准的实现文档。让文档去重复代码细节，就是腐坏之源。
- 真正不会被代码替代的、必须由文档承载的，只有三类：**需求为什么做（需求交付物）、方案为什么这样设计（设计文档）、系统整体什么样（架构文档）**。
- 除了这三类，其他冗余文档一律不写。

**为什么是"模板"而不是"固定结构"？**
- 模板提供最小必需骨架，不强制字段。
- 每次使用时，按项目实际情况选用、裁剪。
- 模板本身会随实践演进。

---

## 何时加载此 skill

- `planner` agent 启动时（准备输出设计文档）
- 用户明确要求"写需求文档"、"写设计文档"、"写架构文档"
- 新项目初始化、旧项目文档补完

## 模板 1：需求交付物

> 作用：锁定"做什么"和"为什么做"，是后续所有变更的源头。
> 更新时机：需求变化时。代码变化不更新它。

```markdown
# 需求：[一句话标题]

## 背景
[2-3 句话：什么场景、什么用户、当前痛点是什么]

## 目标
[这次要解决什么问题。不写"如何解决"。]

## 非目标（明确不做的）
- [避免被误解为要做的事]
- [避免范围蔓延]

## 用户故事 / 关键场景
1. 作为 [角色]，我希望 [行为]，以便 [收益]
2. ...

## 验收标准（可测）
- [ ] [具体可验证的结果 1]
- [ ] [具体可验证的结果 2]
- [ ] 边界/异常：[例：超长输入、并发冲突如何处理]

## 约束
- 时间：[deadline]
- 技术：[必须用/不能用的栈]
- 合规：[若涉及]

## 未决问题（谋而后动）
- [ ] [需要与 X 确认的点]

<!-- 版本：v1 · 日期：YYYY-MM-DD · 作者：[姓名] -->
```

## 模板 2：设计文档（每次需求变更配一份）

> 作用：锁定"怎么做"的关键决策。
> 更新时机：设计方案确定时写；方案改了就更新或新建版本。
> **不写**：逐行代码说明、工具函数清单、可通过 `git log` 查到的历史。

```markdown
# 设计：[对应的需求标题]

## 关联需求
[需求文档链接或路径]

## 方案摘要
[3-5 句话讲清楚：这次打算怎么做、为什么这样做]

## 关键决策与取舍
| 决策点 | 选择 | 候选方案 | 选择理由 |
|-------|------|---------|---------|
| [例：存储方式] | PostgreSQL | Redis / MongoDB | 强一致性 + 已有基建 |
| ... | | | |

## 涉及模块
- [模块 A]：[它的变更是什么]
- [模块 B]：[它的变更是什么]

## 数据流 / 时序（如适用）
```
[简单 ASCII 流程图或编号步骤]
```

## 接口变更（如有）
- [新增/修改/废弃的接口签名]

## 风险与缓解
- **风险**：[描述]
  - 缓解：[怎么应对]

## 拆分与 Commit 规划
（供 coding 阶段拆步骤用）
1. [ ] Commit 1：[最小可合并单元]
2. [ ] Commit 2：...

## 验收标准（承接需求，可能有细化）
- [ ] ...

<!-- 版本：v1 · 日期：YYYY-MM-DD · 对应 plan session：[session 标识] -->
```

## 模板 3：架构文档（项目级，一份，持续维护）

> 作用：让新人 / 新 session 的 AI 能在 10 分钟内建立对系统的整体认知。
> 更新时机：架构级变更时（新增一层、拆分服务、换存储）。**普通功能开发不更新**。
> **目标长度**：2 页以内。超过就说明讲得太细了，拆细节去设计文档。

```markdown
# 架构：[项目名]

## 一句话定位
[这个系统是什么，为谁服务]

## 技术栈
- 语言 / 运行时：
- 主要框架：
- 存储：
- 关键外部依赖：

## 分层
[用最简洁方式画层级。单向依赖规则写清楚。]

例：
```
UI ─▶ Controller ─▶ Service ─▶ Repository ─▶ DB
横切（Auth / Logging / Metrics）通过 Provider 注入，禁止跨层直接引用。
```

## 核心目录
| 目录 | 职责 | 禁止做的事 |
|------|------|----------|
| `src/api/` | 对外接口 | 不直接访问 DB |
| `src/service/` | 业务逻辑 | 不处理 HTTP |
| `legacy/` | 旧系统代码 | 不要动（除非专项） |

## 关键模式与约定
- [例：所有 API 返回值必须过 zod schema]
- [例：DB 操作只能在 Repository 层]
- [例：不允许跨服务直接调用，通过 event bus]

## 已知技术债（知道在、暂不修）
- [债务 A]：[影响范围 + 暂不修的理由]

## 环境与运行
```bash
./init.sh    # 一键搭环境
npm run dev  # 启动
npm test     # 测试
```

<!-- 最后更新：YYYY-MM-DD · 更新原因：[什么结构性变化] -->
```

## 使用纪律

1. **写模板之前先问自己**：这个字段我真能准确填吗？填不准的不写（宁缺毋滥）。
2. **过时文档比没有文档更危险**。定期清理（Sprint retro 时顺手过一遍）。
3. **AI 不能自主改动架构文档**。AI 可起草，人工必审。
4. **设计文档是一次性的**（对应一次需求），不要把多次迭代都塞进同一份。
