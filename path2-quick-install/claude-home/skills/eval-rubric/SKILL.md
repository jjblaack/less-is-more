# Eval Rubric · 评分维度准则

## 设计思想

**为什么需要 rubric？**
- 如果让 Evaluator 自由发挥"觉得实现得如何"，它会倾向于说"看起来不错"——这是 AI 的已知偏差。
- Rubric 把评估从"主观印象"转成"对照清单逐项打分"，显著降低自评偏差。
- Rubric 是**活的**：发现评估漏掉了某类问题，就补一条维度；某维度长期没用就删掉。

**为什么不写具体项目规范？**
- 本 rubric 是**全局通用骨架**，定义评什么、怎么评。
- 项目特定的评分项（例：必须用 zod / 必须过 E2E）由项目级 CLAUDE.md 或 eval-rubric 补充。

---

## 六大评分维度

### 1. 功能正确性（Critical · 不通过直接打回）
- [ ] plan 文档中的"验收标准"每一条都通过（逐条核对 test_steps）
- [ ] 没有回归：原有的关键测试仍然通过
- [ ] 边界情况处理：空输入 / 超长输入 / 并发 / 异常分支

### 2. 安全性（Critical · 不通过直接打回）
- [ ] 无硬编码 secret / 密码 / token
- [ ] 所有用户输入有验证与边界检查
- [ ] SQL 查询参数化（禁止字符串拼接）
- [ ] 认证/鉴权检查没遗漏关键路由
- [ ] 敏感数据没被日志输出
- 严重漏洞 → 拉 `security-reviewer` subagent 做深度扫

### 3. 边界与错误处理（Major · 扣分）
- [ ] 所有可能失败的 I/O 都有错误处理（不要空 catch）
- [ ] 异常有明确的传播路径或兜底
- [ ] 外部调用有超时配置

### 4. 代码质量与可维护性（Major · 扣分）
- [ ] 无遗留的 console.log / debug 打印
- [ ] 无未用引入 / 死代码 / 注释掉的代码
- [ ] 无绕过类型检查的 `any` / `@ts-ignore` / `# type: ignore`（除非有充分理由并注释）
- [ ] 函数规模合理（无 50+ 行巨型函数、无 4+ 层深嵌套）
- [ ] 命名清晰，与代码库风格一致

### 5. 风格一致性（Minor · 建议）
- [ ] 与代码库既有风格一致（命名、格式、模式）
- [ ] 遵循项目 CLAUDE.md 和架构文档约定

### 6. 性能隐患（Minor · 信息性）
- [ ] 无明显的 N+1 查询
- [ ] 无在循环里做同步 I/O
- [ ] 无显著的 O(n²) 可避免为 O(n)

---

## 评级标准

| 等级 | 触发 | 处置 |
|-----|------|-----|
| **BLOCK** | 任一 Critical 维度有问题 | 打回给 `coder`，必须修复 |
| **WARN** | 无 Critical，但有 2+ Major | 打回，讨论后决定是修还是记技术债 |
| **PASS-WITH-NOTES** | 无 Critical，≤1 Major，有 Minor | 通过，记录 Minor 作为后续优化 |
| **PASS** | 全部通过 | 通过，标记 feature 为 done |

## 输出格式（Evaluator 必须按此格式返回）

```markdown
# 评估报告：[功能标题]

## 总评
[PASS / PASS-WITH-NOTES / WARN / BLOCK]

## 维度打分
| 维度 | 结果 | 问题数 |
|------|------|-------|
| 功能正确性 | ✅/❌ | 0 |
| 安全性 | ✅/❌ | 0 |
| 边界与错误处理 | ✅/⚠️ | 0 |
| 代码质量 | ✅/⚠️ | 0 |
| 风格一致性 | ✅/ℹ️ | 0 |
| 性能 | ✅/ℹ️ | 0 |

## 详细问题清单
### Critical
- [问题描述 + 文件:行号 + 修复建议]

### Major
- ...

### Minor
- ...

## Commit 评估文档（git commit 风格建议）
- 建议的 commit message 调整：...
- 拆分建议：该 commit 是否应进一步拆成 N 个？

## 结论
[一段话总结：通过还是打回，下一步做什么]
```

## 评估纪律

1. **独立 context**：Evaluator 必须在新 session 启动，不得与 coding 共 context。
2. **不读实现思路**：只看代码和验收标准，不读 coder 的自述和解释。
3. **找问题优先**：Evaluator 的默认立场是"找问题"，不是"找亮点"。宁可误报请人判断，不可漏报让 Critical 漏过。
4. **实际运行测试**：可执行的 test_steps 必须实际跑，不能只看"代码看起来会通过"。

## Rubric 演进

发现评估漏掉的类别（例如"多次出现并发竞态但 rubric 没覆盖"）时：
1. 在本 skill 里加一条维度或 checklist
2. 在 `/harness-retro` 复盘时说明补充原因
3. 标注日期方便追溯
