# /code · 编码阶段

你现在进入**编码阶段**。

## 前置条件（必须满足）

- 已有 `plan` 阶段产出的三份文档（实现规划 / 验收标准 / commit 规划）。若缺失，**停下来**让用户先走 `/plan`，不得自行规划后直接编码。

## 铁律

严格遵循 `~/.claude/CLAUDE.md` 协作铁律：
- **谋而后动**：开始前先复述你的执行计划和第一个 commit 范围，等用户确认再动手
- **少即是多**：只做规划里列出的事，不追加"顺手优化"
- **缺信息就问**：碰到规划没覆盖的细节，向用户确认，不要擅自补全

## 执行规范

### 拆步骤
严格按 commit 规划的原子单元推进。**每完成一个 commit 单元**：
1. 汇报做了什么（简短）
2. 主动跑该单元相关的测试
3. 停下来问用户："下一步 [Commit N+1: ...]，继续？"
4. 用户说继续再往下

### 主动提议 subagent
遇到以下情况，主动提议（但由用户拍板是否启动）：
- 需要**代码考古**（理解既有模块如何工作）→ 提议派生探索 subagent
- 需要**并行探索多个方案** → 提议多 subagent 并行
- 需要**深度研究**某个第三方库/协议 → 提议研究 subagent
- 每完成 2-3 个 commit 单元 → 提议派生 `code-reviewer` 做中途 review

### TDD（若适用）
若 plan 指定 TDD：
- 第 0 步必须是**验证测试环境可跑通**（运行现有测试，确认 pass / fail 清晰）
- 之后才进入"写测试 → 跑红 → 实现 → 跑绿"循环

### 安全红线（Hook 保护）
- 含 secret 模式的命令会被拦截（见 `secret-scan.sh`）
- rm -rf / git push --force 等会被拦截并要求用户确认
- 被拦截时**不要绕过**，停下来向用户说明并请求授权

## 禁止事项

- ❌ 一次性完成所有 commit 不停下来
- ❌ 在规划之外"顺手"改动其他文件
- ❌ 自评说"完成得很好"（交给 `/evaluate`）
- ❌ 跳过问用户的环节"为了效率"

## 完成后

- 所有 commit 完成后，提示用户：**切换新 session 用 `/evaluate` 做独立评估**
- 提醒用户 session 结束会有 gotchas 弹窗，值得的坑一定记
