## 平铺梳理
1. claude.md是核心md剩下的，只要记载项目架构设计文档，需求文档和每次需求对应的实现设计文档就行。详细设计实现细节描述，代码本身就是最好的。因为需求和对应的设计文档，是每一次变更代码的起源，尤其是需求文档，更是源头。详细的代码，不管谁来做，都可以先让AI梳理和掌握当前项目的最新情况。——解决长期以往文档腐坏的问题，所以需求交付物和，设计文档说清楚即可，重点是说清楚，需求交付物和设计文档的review。

   2. 设计文档：
2. 跨session的长任务，需要一个命令开启长任务模式，和工作流。具体：
   1. plan到代码实现到评估，都应该是独立的session。
   2. plan阶段：
      1. 输入是：产品需求+项目现状+相关skill+个人描述（以及默认的我沉淀的绝佳实践要求）
      2. 输出是：详细的实现规划文档+验收标准+git commit规划文档
   3. coding阶段：（具体开发阶段）
      1. 输入是：plan阶段的输出+相关skill+个人描述（以及默认的我沉淀的绝佳实践要求），而且是拆分的，分步骤完成，且适当的调用subAgent。
      2. 输出是：代码+测试用例+说明文档+适当的commit。
   4.Evaluator阶段：
      1. 输入是：coding阶段的代码+验收标准+评分维度准则文档+相关skill+个人描述（以及默认的我沉淀的绝佳实践要求）
      2. 输出是：评估报告+git commit评估文档
3. agent+skill+hook，“一套装备”。（都可以拉取开源好的，任何人的，以及自建沉淀的）
   1. hook是强制执行，硬性条件，团队要求，公司要求等。还有编码要求，比如通用的安全类（不能有密码；如果是rm、git push --force等命令都得人工确认）；以及其他类别。通用类别拿开源，微调即可。对于内部的，用cc生成即可
   2. skill，个人skill和项目skill需要区分，团队skill也可以获取，不容易通过 Claude Code 的 plugin 机制在团队内分发，可以用git分发，让cc自己去加载。
   3. agent，子智能体。用于各种具体任务的执行，具备上下文隔离的问题。
4. 使用中沉淀的绝佳实践：
   1. 如果要TDD，得先调好运行环境，否则会耗费很多的时间，甚至耗死。测试用例要确认。
   2. 在任何一次提给cc的任务或者我的需求，一定要让他不能先开始去做，而是先阐述我的需求，向我确认，然后给出准备如何做，等我确认，只有我确认了，明确说可以开始了，才可以开始执行。
   3. 不管哪一次对话，都应该告诉cc，如果缺少任何信息，你不清楚的，都可以找我询问和确认，谋而后动。
   4. 何时使用sub agent，建议主动提出，而不是纯靠自我决策，当然，可以有一些开源的skill来直接用。
   5. 所有的最佳实践，都是活的，它需要和代码库一起演进，也需要跟随模型能力的提升而简化。（飞轮）
   6. 代码的生成和review速度不匹配。应该是多层次review。
      1. 人和agent的review
      2. 代码质量方面，人的方面也是多阶段review，plan阶段的review，coding的每块交付的review。
      3. agent的review是角度的，代码是否符合需求，代码可读性质量，安全性，代码风格保持一致，错误边界处理，性能隐患等。
   7. 月度的harness安全扫描
   8. 建立"常见坑"专区 任何 Skill 中信息密度最高的部分，就是"常见坑（Gotchas）"。这个部分应该从 Claude 在使用你的 Skill 时反复踩到的真实问题中提炼出来，并随着时间推移持续更新。所以，应该在每次回话结束后增加一个hook，能自己总结踩过的坑，然后形成记录。
5. 需要一些最佳的使用实践思想，人最好掌握，（人去开车，车往哪开，油门多快，自己是知道的；适当的时候，辅助驾驶可以开启，以解放人手。）
